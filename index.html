Font Loading Strategies

Simple @font-face

Throw a naked @font-face block on your page and hope for the best.

Pros

  • Very simple
  • Very future friendly
  • Good rendering performance in Internet Explorer and Edge
  • Does not require modification of the fonts

Cons

  • Bad rendering performance everywhere else
  • Not very robust, yet
  • No easy way to group requests or repaints together

font-display

Add a new font-display: swap descriptor to your @font-face block to opt-in to FOUT on browsers that support it. Optionally, font-display: fallback or font-display: optional can be used if you consider web fonts to be unnecessary to the design.

Pros

  • Very simple
  • Good rendering performance
  • Super future friendly
  • Very robust
  • Does not require modification of the fonts

Cons

  • Only available on Chrome (version 60+ on Desktop and Android)
  • Limited flexibility
  • Hosting

Preload

Add <link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin> to fetch your font sooner. Pairs nicely with an unceremonious @font-face block and feel free to also throw in the font-display descriptor as well for bonus points.

Pros

  • Super easy to implement, one <link> and you’re off
  • Better rendering performance than a @font-face block
  • Future friendly if you use the type attribute to specify the font format
  • Does not require modification of the fonts

Cons

  • Scalability
  • Limited browser support
  • Flexibility
  • You probably wouldn’t be able to use this with a third party host

Inline Data URI

There are typically two kinds of inlining covered by this method: in a blocking <link rel="stylesheet"> request or in a <style> element in the server rendered markup

Pros

  • Seemingly great rendering performance
  • Flexibility
  • Robustness

Cons

  • A catch with rendering performance
  • Browser support
  • Bad scalability
  • Self hosting

Asynchronous Data URI Stylesheetd

Use a tool like loadCSS to fetch a stylesheet with all of the fonts embedded as Data URIs

Pros

  • Rendering performance
  • Flexibility
  • Ease: Does not require any additional CSS changes to use
  • Robust: If the asynchronous request fails, fallback text continues to be shown

Cons

  • Rendering performance
  • Flexibility and Scalability
  • Not maintenance friendly
  • Browser support
  • Self Hosting

Don’t use Web Fonts

It isn’t technically a font loading strategy. But it’s better than using web fonts incorrectly.

Pros

  • Not sure it could be simpler: just use font-family without @font-face
  • Near instant rendering performance

Cons

  • Limited availability
